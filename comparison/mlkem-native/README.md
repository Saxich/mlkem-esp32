# ML-KEM on ESP32 with mlkem-native (ESP-IDF v6.0)

Benchmark of ML-KEM-512/768/1024 (FIPS 203) on ESP32 using mlkem-native v1.0.0 from [pq-code-package](https://github.com/pq-code-package/mlkem-native).

## Installation

### 1. Place project files

Copy the contents of the `test-native` folder into your desired working directory:

```bash
cp -r test-native/ /path/to/your/destination/
cd /path/to/your/destination/
```

> The folder may be renamed — what matters is that `main/`, `components/`, `CMakeLists.txt`, and `setup_mlkem-native.sh` are all present at the root level.

### 2. Download mlkem-native

```bash
bash setup_mlkem-native.sh
```

This will:
- Clone mlkem-native v1.0.0 from pq-code-package into a temporary directory
- Copy the mlkem source tree into `components/mlkem_native/`
- Generate `components/mlkem_native/randombytes.c` with an ESP32 implementation using `esp_fill_random()`
- Generate `components/mlkem_native/CMakeLists.txt` for ESP-IDF

### 3. Set the ESP32 target

```bash
idf.py set-target esp32
```

### 4. Configure optimization level (important for accurate benchmarks)

```bash
idf.py menuconfig
```

Navigate to:

```
Component config
  → Compiler options
    → Optimization Level
      → (●) Optimize for performance (-O2)
```

> **Default is "Debug" (`-Og`) which is significantly slower than `-O2`.**
> Always use "Optimize for performance" when running benchmarks.

Save and exit (`S` → `Q`).

### 5. Select ML-KEM security level

The parameter set is configured in a single place — the root [CMakeLists.txt](CMakeLists.txt):

```cmake
# ML-KEM security level: 512, 768, or 1024.
set(MLKEM_PARAMETER_SET 1024)
```

Changing this value automatically propagates to both the internal build macros and the public API namespace prefix. After changing, delete the build directory and rebuild:

```bash
rm -rf build/
idf.py build
```

### 6. Build

```bash
idf.py build
```

### 7. Flash and monitor

```bash
idf.py flash monitor -p COMx          # Windows (e.g. COM5)
idf.py flash monitor -p /dev/ttyUSB0  # Linux/macOS
```

## What the benchmark measures

The firmware runs three test suites automatically on boot:

**Memory benchmark** (100 iterations each):
- Stack peak usage — measured via `uxTaskGetStackHighWaterMark()` in a dedicated FreeRTOS task
- Heap peak usage — measured via `heap_caps_monitor_local_minimum_free_size_start/stop()`
- Each operation (KeyGen, Encapsulation, Decapsulation) runs in its own task of size `MEM_TASK_SIZE = 40000` bytes

**Performance benchmark** (200 iterations each):
- Cycle counts measured with `esp_cpu_get_cycle_count()`
- Preceded by 10 warm-up iterations

**Integrity check** (100 iterations):
- Full KeyGen → Encapsulation → Decapsulation round trip
- Verifies that shared secrets match: `ss_enc == ss_dec`

## Component structure

```
components/mlkem_native/
├── CMakeLists.txt          ← generated by setup_mlkem-native.sh
├── mlkem_native.c          ← Single Compilation Unit (SCU), includes all src/*.c
├── mlkem_native.h          ← public API
├── mlkem_native.S          ← assembly stubs (not used on xtensa)
├── randombytes.c           ← generated by setup_mlkem-native.sh, uses esp_fill_random()
└── src/                    ← mlkem-native source tree (copied from upstream)
```

> `mlkem_native.c` is an auto-generated SCU that `#include`s all source files. Only it and `randombytes.c` are compiled — do not add the individual `src/*.c` files to the build.
