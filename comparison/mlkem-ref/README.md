# Kyber on ESP32 with pq-crystals/kyber ref (ESP-IDF v6.0)

Benchmark of Kyber512/768/1024 on ESP32 using the reference C implementation from [pq-crystals/kyber](https://github.com/pq-crystals/kyber).

## Prerequisites

- [ESP-IDF v6.0](https://docs.espressif.com/projects/esp-idf/en/v6.0/esp32/get-started/index.html) installed and activated (`idf.py` in PATH)

## Installation

### 1. Place project files

Copy the contents of the `ref` folder into your desired working directory:

```bash
cp -r ref/ /path/to/your/destination/
cd /path/to/your/destination/
```

> The folder may be renamed — what matters is that `main/`, `CMakeLists.txt`, and `setup_kyber_ref.sh` are all present at the root level.

### 2. Download kyber/ref sources

```bash
bash setup_kyber_ref.sh
```

This will:
- Clone pq-crystals/kyber at a pinned commit into a temporary directory
- Copy the `ref/` source tree into `components/kyber_ref/`
- Generate `components/kyber_ref/randombytes.c` with an ESP32 implementation using `esp_fill_random()`
- Generate `components/kyber_ref/CMakeLists.txt` for ESP-IDF

### 3. Set the ESP32 target

```bash
idf.py set-target esp32
```

### 4. Configure menuconfig (optional)

```bash
idf.py menuconfig
```

**Optimization level** *(TODO: decide which)*

```
Component config
  → Compiler options
    → Optimization Level
      → (●) Optimize for performance (-O2)
```

> Default is "Debug" (`-Og`). You may switch to `-O2`, but results at `-Og` are still valid benchmarks.

**CPU frequency** *(optional)*

```
(Top)
  → Component config
    → ESP System Settings
      → CPU frequency
```

Save and exit (`S` → `Q`).

### 5. Select Kyber security level

The parameter set is configured in a single place — the root [CMakeLists.txt](CMakeLists.txt):

```cmake
# Kyber security level: 2=Kyber512, 3=Kyber768, 4=Kyber1024
set(KYBER_K_VALUE 3)
```

After changing, delete the build directory and rebuild:

```bash
rm -rf build/
idf.py build
```

### 6. Build

```bash
idf.py build
```

### 7. Flash and monitor

```bash
idf.py flash monitor -p COMx          # Windows (e.g. COM5)
idf.py flash monitor -p /dev/ttyUSB0  # Linux/macOS
```

## What the benchmark measures

The firmware runs three test suites automatically on boot:

**Memory benchmark** (100 iterations each):
- Stack peak usage — measured via `uxTaskGetStackHighWaterMark()` in a dedicated FreeRTOS task
- Heap peak usage — measured via `heap_caps_monitor_local_minimum_free_size_start/stop()`
- Each operation (KeyGen, Encapsulation, Decapsulation) runs in its own task of size `MEM_TASK_SIZE = 40000` bytes

**Performance benchmark** (200 iterations each):
- Cycle counts measured with `esp_cpu_get_cycle_count()`
- Preceded by 10 warm-up iterations

**Integrity check** (100 iterations):
- Full KeyGen → Encapsulation → Decapsulation round trip
- Verifies that shared secrets match: `ss_enc == ss_dec`

## Component structure

```
components/kyber_ref/
├── CMakeLists.txt          ← generated by setup_kyber_ref.sh
├── randombytes.c           ← generated by setup_kyber_ref.sh, uses esp_fill_random()
├── kem.c / kem.h           ← KEM API (crypto_kem_keypair / enc / dec)
├── indcpa.c / indcpa.h     ← IND-CPA core
├── polyvec.c / poly.c      ← polynomial arithmetic
├── ntt.c / reduce.c        ← NTT and modular reduction
├── cbd.c / verify.c        ← sampling and constant-time comparison
├── fips202.c / fips202.h   ← SHA-3 / SHAKE (Keccak)
├── symmetric-shake.c       ← SHAKE-based PRF/KDF wrappers
└── params.h / api.h        ← compile-time parameters and size constants
```

> Each `.c` file is compiled individually. `KYBER_K` is injected at compile time via `target_compile_definitions` and controls which parameter set is active.
