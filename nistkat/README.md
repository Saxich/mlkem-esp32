# NIST KAT Verification Directory

This directory contains Known Answer Test (KAT) vectors generated for the ESP32 microcontroller, covering all three ML-KEM security levels: **ML-KEM-512**, **ML-KEM-768**, and **ML-KEM-1024**. Each security level includes 1000 input vectors and 1000 corresponding hashed output vectors.

## Background

When optimizing the ML-KEM implementation for ESP32, it was necessary to repeatedly verify that code changes did not compromise the correctness of the cryptosystem. For this purpose, a KAT test was implemented — a standard method where known input vectors must produce verified output vectors.

The reference implementation includes its own KAT test, but its use is limited: it relies on PRNG functions, which our implementation intentionally replaces with TRNG, and it provides only 100 test vectors. We therefore designed a custom KAT test.

As test vectors, we used the vectors from the KAT project by Kris Kwiatkowski, which provides 1000 sets of vectors for each ML-KEM security level. Unlike the reference vectors, each set contains not only a PRNG seed but also values pre-generated by `randombytes()`. Random value generation can thus be bypassed entirely by calling `crypto_kem_*_derand()` and passing the pre-generated coins directly. The same input vectors are shared across all three security levels — only the output vectors differ between them.

### Memory Optimization

The ESP32 has 4 MB of flash memory. Storing 1000 raw output vectors for ML-KEM-1024 would require approximately 6.336 MB for output data alone (*pk* = 1568 B, *sk* = 3168 B, *ct* = 1568 B, *ss* = 32 B, sum × 1000). To fit within device constraints, output vectors are stored as hashes. Each output set is hashed with `SHA3_256()` and only the 32-byte digest is stored in flash. This reduces storage to 1000 × 32 × 4 = 128,000 B (≈ 0.128 MB), making it feasible to verify all vectors on the device.

Using 1000 vectors proved to be the right decision. In isolated cases during development, race conditions or incorrect rounding caused wrong outputs in only a few vectors out of a thousand — discrepancies that would likely have gone undetected with a smaller vector set.

## Validity Proof via Reference Implementation

This directory is structured as a drop-in replacement for the `nistkat` directory in the [pq-crystals/kyber](https://github.com/pq-crystals/kyber/tree/main/ref/nistkat) reference implementation. Running our vectors through the reference implementation serves as an independent proof of their correctness.

### Steps

1. Clone the reference implementation:
   ```bash
   git clone https://github.com/pq-crystals/kyber.git
   cd kyber/ref
   ```

2. Replace the original `nistkat` directory with this one:
   ```bash
   rm -rf nistkat
   cp -r /path/to/this/nistkat ./nistkat
   ```

3. Compile:
   ```bash
   make nistkat
   ```

4. Run the test binary for the desired security level:
   ```bash
   ./nistkat/PQCgenKAT_kem512
   ./nistkat/PQCgenKAT_kem768
   ./nistkat/PQCgenKAT_kem1024
   ```

A clean run with no errors confirms that our test vectors are valid and that the KAT test used on the microcontroller is correct.
